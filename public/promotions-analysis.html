Cart rules vs Item rules
🛒 Cart rules
Step counting and discount distribution based on the whole cart. Discount split proportionally among all items.
📦 Item rules
Step counting and discount applied per specified item independently. Discount on each matching product separately.

Fake Rules — Promotional Text Workaround

🎭 Workaround Pattern4 / 11
What is a fake rule?
A fake rule is a Cart Price Rule configured with a 0% discount value — not to apply any discount, but purely to attach promotional text, action labels, and frontend visibility to a promotion whose actual pricing is handled elsewhere (e.g. Advanced Pricing).
🧮
Magento's Advanced Pricing module, when active, prevents attaching promotional text fields to standard cart rules in certain configurations. The workaround: create a shadow 0% rule to carry the text payload.
Why teams use this
🏷️
Promotional banners & product labels require a cart rule with "Visible on frontend" enabled. Advanced Pricing has no equivalent field.
📄
Promo title, text, legal disclaimer — these fields exist on cart rules only. When a pricing override handles the discount, a fake rule is the only vehicle for this content.
🎯
Action labels on product tiles must reflect the promotion type. These attach to cart rules, not advanced pricing entries.
Use cases — when fake rules appear
🏷️
Labels SKCZPLROIT — Display action labels on product & category pages (e.g. "Black Friday", expiration flags).
📄
Display description ITRO — Show promotion description in checkout. RO: mandatory legal text when advanced pricing is active.
⚖️
Omnibus compliance RO — Attach label showing the lowest price in the last 30 days (legally required by EU Omnibus directive).
📦
Group products for landing pages RO — Filter & group products by salesRuleId for campaign collections (e.g. "Black Friday deals").
⚡
Quick pricing changes RO — Competitive reaction, supplier-negotiated deals & internal promos via rule instead of direct price change.
📊
Rule tracking RO — Using fake rules as a tracking mechanism for campaign attribution and reporting workaround.
Impact & new system requirement
⚠️
Inflates rule count — stale fake rules accumulate without cleanup tooling. No visual distinction between real and fake rules in Magento UI.
😵
Cognitive overhead — rule managers must remember which rules are "real" vs. text-only carriers. Risk of confusion and accidental misuse.
✅
New system requirement: Promotional text, labels, and banners must be configurable independently of discount value — decoupling eliminates this workaround entirely.

What Works Well

✅ Strengths of Current Solution6 / 11
🔧
Flexible rule engine
34 rule type combinations covering virtually all promotion scenarios. Qty, Spent, Realspent triggers × Cart/Item scope × multiple action types. 26 actively used — the engine is genuinely versatile.
🎟️
Coupon generation & export
Auto-generated unique codes with configurable length, format, prefix/suffix, and CSV export for call centers and partner distribution. Used actively in all 4 workshop markets.
📝
Customer-facing text fields
Promo title, promo text, and legal disclaimer fields embedded in the rule. All markets use these. Legal is mandatory in RO. Supports the full customer communication flow from within the promotion itself.
🏪
Channel targeting
Rules can target specific channels: web, mobile app, brick & mortar. PL uses shop_type conditions to differentiate pharmacy vs online promotions. IT uses Reservation Allowed for Click & Collect.
⚙️
Rich condition system
18 distinct condition attributes used globally. Supports both quantifiable (≤ ≥ = range) and non-quantifiable (include/exclude/AND/OR) operators. Multiple conditions per rule with full logical grouping.
📅
Auto-apply with date validity
Rules auto-apply when conditions are met — no coupon code needed for standard promotions. Date-based from/to scheduling allows advance setup and automatic deactivation.
🛑
Global usage cap
Group ID Max Count and Global Max Usage fields provide budget protection — rules stop firing once a global usage limit is hit. IT and RO use this actively for managed campaign budgets.

Pain Points & Workarounds

🔍 Workshop Findings7 / 11
All markets
📊
Reporting — No in-app reporting on promotion performance. Teams lack visibility into rule effectiveness across campaigns.
✏️
Order edit — Promotions not re-evaluated on order edits. Discount corrections require manual workarounds.
🔗
Non-combinable rules UX — Configuring which rules can co-exist is cumbersome with no clear overview of combination policies.
🎯
Priorities system — Not trusted. 3 of 4 markets use it inconsistently or avoid it entirely.
Poland PL
📋
Order detail info — Insufficient promotion detail on order pages for post-sale review and support.
🔧
Conditions UX — Condition builder is complex and hard to use for non-technical rule managers.
📤
Export missing discount amount — Rule/order export doesn't include applied discount amount per line.
Slovakia SK
🐢
Slow loading of admin — Magento admin performance degrades when many rules are active simultaneously.
🔌
Promotool integration missing — No integration between the promotions module and Promotool planning system.
Italy IT
🧮
Total amount without VAT — Subtotal condition excludes VAT. IT needs VAT-inclusive cart total for accurate thresholds.
🎟️
Farmis voucher — Voucher reconciliation with Farmis ERP is manual and error-prone.
🎁
Multiple gifts — Only one gift item per rule. Multi-gift promotions require multiple rules and manual coordination.
📶
Missing multiple thresholds — Step/tier promotions are limited in how many tiers can be configured per rule.
📊
Volume discount UX — Configuring tiered/volume discounts is unintuitive with no visual step builder.
Italy + Slovakia ITSK
⚙️
Magento admin complicated — Too many fields, cluttered UI. Non-technical users struggle with the admin interface.
Romania RO
🔗
ERP integration — Gaps in Farmis synchronisation cause reconciliation issues for promotions with financial impact.
✅
Missing validations — Rule configuration lacks guardrails. Invalid combinations can be saved without warnings.

Challenges & Risks

⚠️ System & Migration8 / 11
System risks
🎲
Rule stacking unpredictable. "Discard subsequent rules" is the only combinability control — a global on/off switch. No fine-grained interaction management. PL: 37% of orders have 2+ rules applied simultaneously.
📊
Priority system not trusted. 3 of 4 markets use it inconsistently or avoid it entirely. Discount calculation order is unpredictable when multiple rules match the same cart.
🧹
Rule accumulation. No lifecycle tooling — stale rules build up silently. RO: 5,520 rule instances with no automated detection of zero-order rules.
👁️
No pre-activation visibility. Teams cannot see what rules will combine on a given cart before activating a new one. Testing requires creating real orders.
Migration risks
🇸🇰
SK-exclusive rule types. gift_from_qty_cart_max, gift_extra_realspent_cart, and by_fixed_per_qty are used only in SK — may be local Magento customizations. Technical confirmation required before migration scope is set.
🇨🇿
CZ Marketplace integration. mp_fixed_spent_cart handles ~250K orders not present in any other market. Architecture unclear — must be understood before redesign.
🇷🇴
RO data quality. 5,520 rule instances including stale and fake rules. Requires cleanup strategy and stakeholder sign-off before migration.
🇨🇿
CZ workshop gap. Qualitative data not yet collected. Architecture decisions made before CZ workshop may need revisiting.
🛒
Accounting marketplace (SK → all markets). Different marketplace types require distinct financial accounting flows in SK — will soon apply across all markets as marketplace presence expands.
Business risks if not addressed
Discount leakage
Uncontrolled stacking creates unintended discount accumulation — highest risk in PL where 37% of orders already stack multiple rules.
Compliance exposure
Missing VAT-inclusive condition (RO) means thresholds may fire incorrectly. Legal text & ERP fields must carry forward — cannot be dropped in redesign.
Operational overhead
Fake rules, stale rules, and manual workarounds consume team time. Complexity grows with each market — no self-service tooling to clean up.
Scalability ceiling
Current per-market Magento model cannot support global/cross-market promotion campaigns. New architecture must rethink country customisation scope.

Legal & ERP Constraints

⚖️ Compliance Requirements9 / 11
📌
These requirements are non-negotiable — they must be preserved in any redesign of the Promotions Module. The new system must continue to support all of these out of the box.
ERP Integration — mandatory all markets
🔗
Farmis Voucher ID + Code are mandatory in all 4 workshop markets (IT, RO, SK, PL). Every promotion rule must carry ERP identifiers for accounting reconciliation. Any redesign must preserve this link.
💳
Uses per coupon is a mandatory field in Romania, required for financial control. Other markets use it, but RO enforces it on every rule. Relates to budget/cost accounting for individual campaigns.
📤
Generate & export coupon codes is actively used in all markets — codes are distributed to call centers, partners, and loyalty programs. CSV export capability must be maintained.
🏪
Reservation Allowed is critical for Italy (Click & Collect / pharmacy reservation flow). Rules must be applicable to reservation orders, not just checkout purchases.
Legal requirements
🇵🇱
No free gifts — Poland. Gifts are legally not allowed to be free. Any gift promotion must be structured as a paid item at minimal cost, not 0€.
🇵🇱
Omnichannel sales rule — Poland. Legal obstacle currently blocking omnichannel rule application. New legal rules under investigation to resolve this constraint.
🇷🇴
Global maximal usage display — Romania. Legally required to display to customers how many items/uses remain in a limited-stock promotion.
📄
Promo legal text field is mandatory in Romania by local law — promotional text must include a legal disclaimer. Must remain a first-class field in any redesign.
🏷️
Action labels (product tiles) must accurately represent the discount type. Legally required — not just decorative. Non-combinable rules for specific voucher types must also be preserved.

Open Points — Team Discussion

💬 Unresolved10 / 11
Business decisions
🎟️
Coupons — Where does coupon ownership live? COMS (order management) vs. Promotions module?
🚚
Free shipping — Is free shipping a Checkout concern or a Promotions rule? Impacts rule engine scope.
📁
Imports/exports — Allow bulk rule imports? CSV or API? Should templates or condition presets be supported?
Technical decisions
🏷️
Action labels — Label service integration or source-of-truth in promo rules? Who owns label rendering on product tiles?
📦
Bundles — Where do bundle promotions live? Separate bundle module or within promo engine?
🗂️
Content product grouping — Allow grouping/filtering products by salesRuleId for campaign landing pages?
📍
Placements — Where and how are promotion placements managed? Scope TBD.
💲
Advanced pricing — Coexistence model: keep separate module or have Promotions module cover all pricing scenarios?
📊
Reporting — In-app reporting or Power BI? Level of detail and real-time requirement? Must include number of completed orders per rule.
More technical decisions
🔌
API for third parties — Expose promotions API for external systems? Auth model, rate limits, scope?
⏸️
Schedule pausing vs. multiple date ranges — Countries preferred multiple start/end dates over a single pause mechanism. Which approach wins?
🏪
Reservation allowed — Which order types should promotions apply to? Click & Collect, pharmacy reservations, etc.
Context
These open points were collected during country workshops. Business items require alignment between product owners and country leads. Technical items require architecture decisions before implementation scoping.
⚡
Critical path blockers: Coupons ownership, Free shipping scope, and Action labels source must be resolved before rule creation UX can be finalised.
